## 1. 🔴 Рандомные варианты описаний — `getRandomDescription()`

### В чём проблема

```js
// server/utils/jsonLoader.js
export function getRandomDescription(descriptions) {
  const randomIndex = Math.floor(Math.random() * descriptions.length);
  return descriptions[randomIndex];
}
```

Каждый вызов `getRandomDescription()` выбирает случайный вариант из 6–12 описаний. Это значит:

- **Одни и те же данные рождения → разный текст при каждом обновлении страницы**
- Пользователь видит одну интерпретацию «Солнца в Овне» сейчас, и совсем другую через минуту
- Это разрушает доверие: «какая из них правильная?»

### Когда много вариантов оправданы

- Контекстный выбор (разный тон для разных типов пользователей)
- A/B-тестирование конверсии
- Учёт истории сессий (не повторять то же самое)

**Ничего из этого сейчас не реализовано.** Это чистый `Math.random()` — самый дешёвый и самый разрушительный способ.

### Объём

| Категория | Файлов | Ключей | Вариантов на ключ | Всего описаний |
|-----------|--------|--------|:-----------------:|:--------------:|
| `planet_sign` | 11 | 132 | 6–12 | 1,512 |
| `planet_house` | 11 | 132 | 6–8 | 1,032 |
| `aspects` | 55 | 275 | 5–6 | 1,649 |
| `sign_pair_combos` | 6 | ~864 | 4–6 | 5,180 |
| `life_areas` | 1 | 7 | 12 | 84 |
| **Итого** | **84** | ~1,410 | — | **~9,457** |

### Рекомендация

**Оставить 1 лучший вариант на каждый ключ.** Не удалять остальные — сохранить в архив на случай будущего использования (контекстный выбор, A/B).

#### Вариант A (быстрый, ~2 часа)
1. Для каждого ключа выбрать один «каноничный» вариант (самый полный/качественный)
2. Переименовать текущие JSON в `data/generated/_archive/`
3. Создать новые JSON с одним вариантом на ключ
4. Заменить `getRandomDescription()` на `getDescription()` без рандома
5. **Если хочется детерминированного разнообразия** — хэшировать birth data в seed для выбора

#### Вариант B (правильный, ~1 день)
1. Создать скрипт-валидатор, который для каждого ключа:
   - Сравнивает все варианты по длине, полноте, наличию ключевых терминов
   - Выбирает лучший
   - Логирует дубликаты и слишком короткие варианты
2. Сформировать новые JSON
3. Обновить `jsonLoader.js`

**Планка качества:** для каждой комбинации (планета + знак, планета + дом, пара планет + аспект) должен быть один авторитетный текст. Если хочется нескольких — они должны выбираться детерминированно и осмысленно.

## 2. 🟠 49/55 пар без sign_pair_combos (89% пробел)

### В чём проблема

Из 55 возможных планетарных пар только 6 имеют данные в `sign_pair_combos/`:

```
✅ Есть:    jupiter_saturn, mars_saturn, mercury_venus, moon_venus, sun_moon, sun_venus
❌ Нет (49): jupiter_asc, jupiter_neptune, jupiter_pluto, ... (все остальные)
```

### Как это влияет на пользователя

Алгоритм `getAspectDescription()` идёт по цепочке:

```
sign_combo (exact) → НЕТ ФАЙЛА → aspect (exact) → аспект без знакового контекста
```

Для 89% планетарных пар пользователь **никогда не получает интерпретацию с учётом знаков**, даже если астрологически это важно (например, соединение Солнца и Луны в Овне vs в Рыбах — совершенно разный смысл).

Фактически: интерпретация аспектов для большинства пар — чистый аспект без учёта знаков.

### Рекомендация

**Фаза 1 — приоритезировать:**
- 12 пар с Солнцем (солнце + любая планета): `sun_asc`, `sun_jupiter`, `sun_mars`, `sun_mercury`, `sun_neptune`, `sun_pluto`, `sun_saturn`, `sun_uranus` (sun_moon, sun_venus уже есть)
- 10 пар с Луной: аналогично
- Оставшиеся — по убыванию астрологической значимости

**Фаза 2 — генерировать:**
- Для каждой пары нужно 144 комбинации знаков (12×12), но многие будут астрологически эквивалентны
- Можно сократить до ~12 осмысленных групп (огонь-огонь, огонь-вода, etc.) + особые случаи
- Использовать LLM для первичной генерации с последующей редактурой астролога

**Фаза 3 — добавить fallback получше:**
- Сейчас при отсутствии sign_combo сразу идём в aspect
- Можно добавить промежуточный уровень: интерпретация по стихиям (стихия планеты1 + стихия планеты2) — это всего 16 комбинаций (4×4 стихии)

## 3. 🟠 136 английских описаний в planet_house

### В чём проблема

```
planet_house/neptune.json → все 96 описаний на английском (12 домов × 8 вариантов)
planet_house/pluto.json  → 40 описаний на английском (дома 1-5 × 8 вариантов)
```

Остальные 9 планет — полностью на русском (896 описаний). `planet_sign` — чистый русский (1512).

### Влияние

- Система инсайтов логирует `missing ru content` для этих планет
- Пользователь в русскоязычном приложении видит английский текст или заглушку
- Нептун и Плутон — ключевые планеты для глубинной интерпретации, их отсутствие на русском критично

### Рекомендация

1. Перевести 136 описаний на русский (LLM + редактура астролога)
2. Удалить `lang: "en"` из финальных данных
3. Проверить, что система инсайтов перестала выдавать `missing ru content`

## 4. 🟡 Нет валидации интерпретаций

### В чём проблема

- Нет механизма проверки, что «Солнце в Овне в 7 доме» + «Марс в Раке в 10 доме» + «Солнце в квадрате к Марсу» не противоречат друг другу
- Нет проверки на соответствие астрологическим принципам
- При 9,457 описаниях ручная проверка невозможна

### Рекомендация

**Создать агента-валидатора** (отдельный sub-agent или cron-job), который:
1. Принимает полный ответ API (как тот, что возвращает `astro.get.js`)
2. Проверяет:
   - Нет прямых противоречий между интерпретациями
   - Нет дублирования (разные планеты в одном знаке с одинаковым текстом)
   - Все описания на русском
   - Все ссылки на знаки/дома/аспекты соответствуют вычисленным
3. Логирует проблемы в отчёт
4. Может запускаться при сборке или по расписанию

**Минимальная версия:** скрипт, который дёргает API с тестовыми данными, собирает все description.full и ищет очевидные проблемы (EN-текст, пустые поля, несоответствие `type` фактическому содержанию).

## 5. 🟡 Нет кэширования данных

### В чём проблема

Каждый запрос к `/api/astro`:
- Заново читает десятки JSON-файлов через `fs.readFile`
- Для каждого аспекта проходит fallback-цепочку
- Результат: лишняя latency, лишний I/O

### Рекомендация

- Загружать все JSON при старте сервера в `Map<string, object>`
- `getRandomDescription()` (или его замена) работает с in-memory данными
- При обновлении данных — перезагружать кэш

## Ответ на твой вопрос: «может, логично иметь один вариант?»

**Да, один вариант логичнее.** Вот почему:

1. **Доверие.** Астрологическая консультация — это про авторитетность. Когда один и тот же запрос даёт разные тексты, это подрывает доверие: «система просто выдаёт случайный текст из кучи».

2. **Качество vs количество.** Один хорошо написанный, глубокий текст для «Солнца в Овне» ценнее 12 посредственных. Редактировать и поддерживать 12 вариантов на каждый ключ (1,410 ключей) практически невозможно.

3. **Случайность ≠ персонализация.** Если бы система выбирала вариант на основе других факторов карты — это была бы персонализация. Но `Math.random()` — это не персонализация, это лотерея.

4. **Единственное оправдание для нескольких вариантов** — если они осмысленно разные (разный тон, разный фокус) и выбираются не случайно. Пока этого нет — лучше один качественный.

**Компромисс на будущее:** если захочешь разнообразия, сделай детерминированный выбор на основе хэша birth data. Один и тот же человек всегда будет видеть один и тот же текст, но разные люди увидят разные варианты.
